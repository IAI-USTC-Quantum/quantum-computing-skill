# AGENTS.md

> 本文件为本地 agent 工作说明，**不要纳入版本控制、不要提交到远端**。
> 建议仅在本地保留，并让 `CLAUDE.md` 作为它的软链接存在。

本文件只提供维护这个 skill 仓库所需的最小约束。若与仓库实际内容冲突，以 `README.md`、`SKILL.md`、`references/`、`examples/`、`scripts/` 为准。

## 职责分工

| 文件 | 面向对象 | 应写什么 | 不应写什么 |
|---|---|---|---|
| `AGENTS.md` / `CLAUDE.md` | skill 维护者、本地 agent | 维护约束、本地约定、仓库边界、提交流程、回复格式 | skill 用法、安装策略、工作流、排障细节、其他面向用户的说明 |
| `SKILL.md` / `references/` | 调用此 skill 的 agent（面向用户） | 触发条件、操作规则、导航、工作流、输入输出、限制条件、排障顺序 | skill 维护说明、个人本地配置、与 skill 开发者相关的流程 |
| `README.md` | 准备安装此 skill 的人类 | skill 安装方式、仓库职责、非目标范围 | 详细操作手册、长篇排障、上游内部实现细节 |

## 维护规则

- **先看仓库，再下结论**：先读 `README.md`、`SKILL.md`、`.gitignore`，再看 `references/`、`examples/`、`scripts/`。
- **这是 skill 仓库，不是上游源码仓库**：只维护 skill 的触发条件、说明结构、示例和辅助脚本，不修改 `UnifiedQuantum` 上游代码。
- **保持渐进式披露**：`description` 只写触发条件与核心能力；`SKILL.md` 负责导航；细节下沉到 `references/*.md`。
- **优先复用现有材料**：新增说明、示例或脚本前，先检查 `references/`、`examples/`、`scripts/` 能否复用或小改。
- **内容重在“如何使用”**：优先写调用入口、工作流、输入输出、限制条件和排障顺序；不要搬运上游内部实现、构建测试或发布细节。
- **文档与引用保持一致**：仓库内只写相对路径；改动 `README.md`、`SKILL.md`、`references/`、`examples/`、`scripts/` 时同步更新相关引用与文件名。
- **示例与脚本保持克制**：示例短小、聚焦、可复用；脚本只放值得复用的检查或辅助逻辑。新增 Python 脚本优先 `uv run`，必要时使用 PEP 723。
- **默认不绑定特定宿主工具名**：能写成通用能力描述时，就不要绑定某个 agent 运行环境的专有工具名；只有在能力不可避免依赖宿主时，才明确点名。
- **影响边界时先确认**：如果改动会影响触发范围、目录结构、示例行为或宿主定位，先确认再动手。
- **Git 与文件操作保持谨慎**：未经要求不 `git add`、`git commit`、`git push`；若用户要求提交，先说明当前工作区和本次改动范围；删除优先 `trash-put`。
- **优先用内置编辑与搜索工具**：批量改动前先理解全文结构，不要粗暴整段覆盖。
- **回复格式**：每次回复末尾追加“喝水水中”并配一个好玩的 emoji。

## 校验

- 修改 `SKILL.md` 或 `README.md` 后，检查目录树、文件名和文内链接是否仍然成立。
- 修改 `examples/` 或 `scripts/` 后，至少做最小运行或静态检查，确认说明和实际行为一致。
- 若发现文档与仓库内容不一致，优先修正文档，不要假设以后再补。

## 本地约定

- `AGENTS.md` 仅供本地 agent 使用，不应提交。
- `CLAUDE.md` 可作为指向 `AGENTS.md` 的软链接。
- 若要降低误提交风险，优先用 `.git/info/exclude` 等本地机制。
